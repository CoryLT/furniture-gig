### Logging labor — three doors, one destination
Every path below writes a Labor ledger expense tagged to a `crew_member_id`, so it shows
in the person's payment history and counts toward their 1099 (the `worker_payments` view
is still the one truth).

1. **Past-sale form** (`app/flipper/pipeline/add-past-sale/PastSaleForm.tsx`) — added a
   **Labor cost** box. It's **labor-only on purpose** (materials/fees come from receipts;
   labor is the one cost with no receipt to scan, so it can't double-book). Includes a
   **"Who did you pay?"** picker with an inline **"+ Add a person"** (find-or-create a
   name-only crew member without leaving the form). `add-past-sale/page.tsx` now loads the
   crew list to feed it. On save → `add_piece_expense(..., p_date = sale month)`. On finish
   it redirects to `/flipper/pipeline?sold=m:YYYY-MM&focus=<pieceId>`.
2. **Crew person's page** (`app/flipper/crew/[id]/page.tsx`) — **"Add a payment"** button
   (`app/flipper/crew/AddPaymentCard.tsx`): amount + date + **optional piece** + note →
   `add_labor_payment`. This is the main "I paid Nick $50" entry point.
3. **Edit-entry page** (`app/books/transaction/[id]/page.tsx`) — when an entry is tied to a
   piece, an **"Add labor paid on this piece"** box (`components/books/AddPieceLaborField.tsx`):
   amount + who → `add_labor_payment(p_piece_id)`. Adds a NEW entry; does not touch the sale.

## Gotchas learned / reinforced this session

- **`worker_payments` counts any expense-line txn that has a `crew_member_id`.** A sale has
  no expense line, so tagging a person to a sale does nothing — that's why the edit-page
  crew dropdown is gated to expense entries.
- **`update_entry` does NOT touch `crew_member_id`.** Set it with a direct, owner-scoped
  `transactions.update({crew_member_id})` (same as `app/api/receipts/save`).
- **Native `<select>` popups need a SOLID background** to be readable in dark mode; a
  transparent/tinted panel var isn't enough even with `color-scheme: dark` set.
- **`add_piece_expense` / `add_labor_payment` don't ping `/api/payments/check-1099`** when
  used from the past-sale form / crew page (deliberate — avoids firing historical 1099
  alerts). Payment Records + the person's page still total correctly from `worker_payments`.
- **raw.githubusercontent.com caches hard.** To verify a fresh push, read the file via the
  authenticated GitHub **contents API** (`api.github.com/.../contents/<path>?ref=main`,
  base64-decode) instead of the raw URL.
- Lucide icon availability isn't guaranteed offline — used `Users`/`Plus` (known-present)
  rather than `Merge`.
